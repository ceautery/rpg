# M6 Design Spec — Quality of Life + Speed

**Date:** 2026-06-20
**Status:** Approved for implementation

---

## Scope

Three independent improvements to the existing turn loop, delivered together as Milestone 6:

1. **Inter-PC coordination phase** — party dialogue before action collection
2. **Model selection** — Haiku for fast agent dispatches
3. **Log simplification** — remove HTML render from the turn loop

No new state files. No new agent files. No architectural changes to the blackboard pattern.

---

## 1. Inter-PC Coordination Phase

### Position in the turn loop

Inserted between step 2 (scene setup) and step 3 (action collection gate) in `run.md`. It is optional and skipped cleanly when there is nothing to say.

### New player agent mode: `COORDINATE`

**Input (passed inline in dispatch prompt):**
- Scene prose (same perception packet used for `TAKE_TURN`)
- The PC's own sheet
- Coordination transcript so far this scene (may be empty on first call)

**Output:** A plain in-character string — dialogue only, no JSON wrapper. Short. The character speaks; nothing is decided yet.

**Model:** `haiku` (short conversational response, not a mechanical decision).

### Human + agent party — loop protocol

```
1. Scene presented to human at terminal.
2. Orchestrator prompts: "Before you act — anything to say to [agent PC name(s)]?"
3. If human types anything (non-empty, non-signal):
     a. Print human's words in the Claude UI.
     b. Dispatch each agent PC in COORDINATE mode with:
          - the human's message
          - the coordination transcript so far
     c. Print agent response(s) in the Claude UI.
     d. Append the full exchange to session.md (see log format below).
     e. Prompt human again: "Anything else? (or 'go' to act)"
     f. Go to step 3.
4. If human types a signal word ('go', 'ready', 'done', or blank line):
     proceed to action collection.
5. If human skips entirely on step 2:
     proceed to action collection immediately.
```

Signal words are case-insensitive. The orchestrator should also accept any message that is unambiguously a readiness signal ("let's go", "move in", "enough talk").

### Agent-only party — when to coordinate

The orchestrator decides whether coordination is meaningful before dispatching. Coordinate when any of the following is true:

- First turn of an encounter with two or more enemies
- Room contains a noted hazard (trap, puzzle, obscured enemy)
- Previous turn ended with an unexpected outcome (ambush, PC downed, enemy fled)
- Party has not yet met this enemy type in the session

Skip coordination for: routine movement between cleared rooms, looting after a cleared scene, rest sequences.

When coordination is triggered, the orchestrator dispatches each agent PC in `COORDINATE` mode in initiative order. Agent A speaks; Agent B receives A's words and responds. One round (each agent speaks once). Then action collection opens.

The orchestrator's judgment is a heuristic check in `run.md` — no additional dispatch needed for the decision itself.

### Session log format

```markdown
**Stone and Aldric — before entering:**
Stone: "You take the right when we go in."
Aldric: "I'll need a clear line. Don't get between me and the target."
Stone: "Noted."
```

The block appears immediately before the `## Turn N, Round 1` heading it precedes. The word "before entering" is replaced with a location cue when relevant ("before crossing the bridge", "before the ambush").

## 2. Model Selection

The `Agent` tool accepts a `model` parameter that overrides the agent file's frontmatter at dispatch time. The agent `.md` files keep `model: sonnet` as the fallback. The orchestrator explicitly passes `model: "haiku"` for fast dispatches.

### Dispatch model table

| Agent | Mode | Model | Rationale |
|---|---|---|---|
| player | `COORDINATE` | haiku | Short in-character line; no mechanical reasoning |
| player | `TAKE_TURN` | haiku | Picking one action from a bounded set |
| player | `CREATE_CHARACTER` | sonnet | Personality, backstory, concept — quality matters; happens once |
| player | `REST` | haiku | Simple: how many Hit Dice to spend |
| world-engine | `RESOLVE` | haiku | Running `dice.py` and arithmetic; no creative reasoning |
| world-engine | `GENERATE` | sonnet | Building maps, stat blocks, encounter scaffolds |
| dm | `SCENE_SETUP` | sonnet | Scene invention, NPC voices, spatial design |
| dm | `ADJUDICATE` | sonnet | Tactical ordering, mechanic identification, narration |

### What changes in `run.md`

Each dispatch template gains a `model:` annotation:

```
Player (take turn):
  Use the player subagent. MODE: TAKE_TURN. Model: haiku.
  ...

World-engine (resolve):
  Use the world-engine subagent. MODE: RESOLVE. Model: haiku.
  ...
```

No changes to `dm.md`, `player.md`, or `world-engine.md`.

## 3. Log Simplification

`scripts/render_log.py` remains in the repository. It is a complete, working tool and may be run manually at any time:

```bash
python3 scripts/render_log.py   # writes log/session.html
```

It is removed from the active turn loop. The orchestrator no longer calls it after each turn.

`run.md` log format section gains one line:

> To watch the session live: `tail -f log/session.md`

`log/session.html` is added to `.gitignore` — it is a generated artifact and does not belong in version control.

## Out of scope for M6

- Agent-to-agent coordination across more than one round (keep it cheap)
- Persisting coordination transcripts to a separate file (session.md is sufficient)
- UI changes beyond what the Claude Code terminal already provides
- M7 (campaign pre-generation) and M8 (import/config) — separate specs
